h1. Основы кэширования

<p>Это введение в три типа техники кэширования, предоставленных Rails по умолчанию, без каких-либо посторонних плагинов.</p>
<p>Перед тем, как начать, убедитесь, что <tt>config.action_controller.perform_caching</tt> установлен <tt>true</tt>, если запущен режим development. Этот флаг обычно устанавливается в соответствующем config/environments/*.rb. По умолчанию кэширование отключено для development и test и включено для production.</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
config.action_controller.perform_caching = true
</pre></div>

<h4 id="page-caching">1. Кэширование страницы</h4>
<p>Кэширование страницы это механизм Rails, позволяющий запросу на сгенерированную страницу быть полностью обслуженным вебсервером (т.е. Apache или nginx) в принципе, без прохождения через стек Rails. Очевидно, это очень быстро. К сожалению, это не может быть применено к каждой ситуации (например, к страницам, требующим аутентификации), и, так как вебсервер фактически извлекает файл из файловой системы, придется иметь дело с вопросом времени хранения кэша.</p>
<p>Чтобы включить кэширование страниц, следует использовать метод <tt>caches_page</tt>.</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ProductsController < ActionController

  caches_page :index

  def index
    @products = Products.all
  end

end
</pre></div>
<p>Предположим, имеется контроллер <tt>ProductsController</tt> и экшн <tt>index</tt>, отображающий все продукты. Как только кто-то впервые вызовет products/index, Rails создаст файл, названный <tt>index.html</tt>, и затем вебсервер будет видеть этот файл до того, как он передаст следующий запрос к products/index приложению на Rails.</p>
<p>По умолчанию директория кэша страниц устанавливается в Rails.public_path (которая обычно устанавливается как папка <tt>public</tt>), и это можно настроить, изменив конфигурационную настройку <tt>config.action_controller.page_cache_directory</tt>. Изменения умолчания с <tt>public</tt> поможет избежать конфликт наименований, поскольку вы сможете захотеть разместить другой статичный html в <tt>public</tt>, но изменение этого потребует перенастройку веб сервера, чтобы он знал, откуда обслуживать кэшированные файлы.</p>
<p>Механизм кэширования страниц автоматически добавит расширение <tt>.html</tt> к запросам страниц, не имеющих расширения, чтобы облегчить веб серверу поиск этих страниц, и это можно настроить, изменив конфигурационную настройку <tt>config.action_controller.page_cache_extension</tt>.</p>
<p>Для того, чтобы эта страница прекращала действие, когда добавляется новый продукт, можно изменить наш пример контроллера следующим образом:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ProductsController < ActionController

  caches_page :index

  def index
    @products = Products.all
  end

  def create
    expire_page :action => :index
  end

end
</pre></div>
<p>Если желаете более сложную схему прекращения, можно использовать уборщики кэша для прекращения кэшированных объектов, если ситуация не меняется. Это раскроем в разделе про Уборщики.</p>
<div class="note"><p>Кэширование страницы игнорирует все параметры. Например, <tt>/products?page=1</tt> будет записан в файловую систему как <tt>products.html</tt> без ссылки на параметр <tt>page</tt>. Следовательно, если кто-то позже запросит <tt>/products?page=2</tt>, ему будет возвращена кэшированная первая страница. Будьте осторожнее, когда кэшируете страницу с параметрами <span class="caps">GET</span> в <span class="caps">URL</span>!</p></div>
<div class='info'><p>Кэширование страниц запускается в последующем (after) фильтре. Следовательно, неверные запросы не будут ложные вхождения кэша, пока вы прерываете их. Обычно эту работу делает перенаправление в некоторых предварительных фильтрах, проверяющие предусловия запроса.</p></div>

<h4 id="action-caching">2. Кэширование экшна</h4>
<p>Одной из проблем кэширования страниц является то, что его нельзя использовать для страниц, требующих проверочный код, определяющий, разрешен ли доступ пользователю. И тут на помощь приходит кэширование экшна. Кэширование экшна работает как хэширование страницы, за исключением того, что входящий веб запрос идет от веб сервера в стек Rails и Action Pack, таким образом, до обслуживания кэша могут быть запущены предварительные (before) фильтры. Это позволит использовать аутентификацию и другие ограничения, и в то же время выводит результат из кэшированной копии.</p>
<p>Чистка кэша происходит так же, как и кэша страниц.</p>
<p>Если хотим, чтобы только авторизованные пользователи могли вызывать экшны в <tt>ProductsController</tt>.</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ProductsController < ActionController

  before_filter :authenticate
  caches_action :index

  def index
    @products = Product.all
  end

  def create
    expire_action :action => :index
  end

end
</pre></div>
<p>Также можно использовать <tt>:if</tt> (или <tt>:unless</tt>), чтобы передать Proc, который определяет, когда экшн должен быть кэширован. Также можно использовать <tt>:layout => false</tt>, чтобы кэшировать без макета, таким образом, динамическая информация в макете, такая как имя вошедшего пользователя или предметы на карте, останется незакэшированной. Эта особенность доступна начиная с Rails 2.2.</p>
<p>Можете изменить путь к кэшу экшна по умолчанию, передав опцию <tt>:cache_path</tt>. Это будет передано непосредственно в <tt>ActionCachePath.path_for</tt>.  Это удобно для экшнов с несколькими возможными маршрутами, которые должны кэшироваться различно. Если задан блок, он будет вызван текущим экземпляром контроллера.</p>
<p>Наконец, если используете memcached или Ehcache, можете также передать <tt>:expires_in</tt>. Фактически, все параметры, не используемые <tt>caches_action</tt>, посылаются в лежащее в основе хранилище кэша.</p>
<div class='info'><p>Кэширование страниц запускается в последующем (after) фильтре. Следовательно, неверные запросы не будут ложные вхождения кэша, пока вы прерываете их. Обычно эту работу делает перенаправление в некоторых предварительных фильтрах, проверяющие предусловия запроса.</p></div>

<h4 id="fragment-caching">3. Кэширование фрагмента</h4>
<p>Жить было бы прекрасно, если бы мы могли закэшировать весь контент страницы или экшна и обслуживать с ним всех. К сожалению, динамические веб приложения обычно создают страницы с рядом компонентов, не все из которых имеют сходные характеристики кэширования. Для устранения таких динамически создаваемых страниц, где различные части страниц нуждаются в кэшировании и прекращаются по-разному, Rails предоставляет механизм, названный Кэширование фрагмента.</p>
<p>Кэширование фрагмента позволяет фрагменту логики вьюхи быть обернутым в блок кэша и обслуженным из хранилища кэша для последующего запроса.</p>
<p>Как пример, если хотите показать все заказы, размещенные на веб сайте, в реальном времени и не хотите кэшировать эту часть страницы, но хотите кэшировать часть страницы, отображающей все доступные продукты, можете использовать следующий кусок кода:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
&lt;% Order.find_recent.each do |o| %&gt;
  &lt;%= o.buyer.name %&gt; bought &lt;% o.product.name %&gt;
&lt;% end %&gt;

&lt;% cache do %&gt;
  All available products:
  &lt;% Product.find(:all).each do |p| %&gt;
    &lt;%= link_to p.name, product_url(p) %&gt;
  &lt;% end %&gt;
&lt;% end %&gt;
</pre></div>
<p>Блок cache в нашем примере будет привязан к вызвавшему его экшну и записан в тоже место, как кэш экшна, что означает, что если хотите кэшировать несколько фрагментов на экшн, следует предоставить <tt>action_suffix</tt> в вызове cache:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
&lt;% cache(:action =&gt; 'recent', :action_suffix =&gt; 'all_prods') do %&gt;
  All available products:
</pre></div>
<p>Можете прекратить кэш, используя метод <tt>expire_fragment</tt>, подобно следующему:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
expire_fragment(:controller =&gt; 'products', :action =&gt; 'recent', 
  :action_suffix =&gt; 'all_prods')
</pre></div>
<p>Если не хотите, чтобы блок cache привязывался к вызвавшему его экшну, можете также использовать глобально настроенные фрагменты, вызвав метод <tt>cache</tt> с ключом, следующим образом:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
<% cache('all_available_products') do %>
  All available products:
<% end %>
</pre></div>
<p>Этот фрагмент затем будет доступен во всех экшнах в <tt>ProductsController</tt> c использованием ключа, и может быть прекращен тем же образом:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
expire_fragment('all_available_products')
</pre></div>

<h4 id="sweepers">4. Уборщики (sweepers)</h4>
<p>Уборка кэша это механизм, позволяющий обойти кучу вызовов <tt>expire_{page,action,fragment}</tt> в коде. Это осуществляется с помощью переноса всей работы, требуемой для прекращения кэшированного содержимого, в класс <tt>ActionController::Caching::Sweeper</tt>. Этот класс является обсервером, просматривающим изменения в объекте через колбэки, и когда изменение случается, он прекращает кэши, связанные с этим объектом, в фильтрах around или after.</p>
<p>Продолжая с нашим примером контроллера Product, мы можем переписать его с уборщиком, следующим образом:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ProductSweeper < ActionController::Caching::Sweeper
  observe Product  # Этот уборщик собирается следить за моделью Product

  # Если наш уборщик обнаружит, что Product был создан, вызываем это
  def after_create(product)
    expire_cache_for(product)
  end

  # Если наш уборщик обнаружит, что Product был обновлен, вызываем это
  def after_update(product)
    expire_cache_for(product)
  end

  # Если наш уборщик обнаружит, что Product был удален, вызываем это
  def after_destroy(product)
    expire_cache_for(product)
  end

  private
  def expire_cache_for(product)
    # Прекращает страницу list теперь, когда мы добавили новый продукт
    expire_page(:controller => 'products', :action => 'index')

    # Прекращает фрагмент
    expire_fragment('all_available_products')
  end
end
</pre></div>
<p>Можно отметить, что в уборщик передается фактический product, поэтому, если кэшируем экшн edit для каждого product, можно добавить метод прекращения, определяющий страницу, которая должна быть прекращена:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
  expire_action(:controller => 'products', :action => 'edit', :id => product)
</pre></div>
<p>Затем добавим уборщик к контроллеру, чтобы сказать ему вызвать уборщик при вызове определенных экшнов. Поэтому, если мы хотим прекращать кэшированное содержимое для экшнов list и edit при вызове экшна create, мы должны сделать следующее:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ProductsController < ActionController

  before_filter :authenticate
  caches_action :index
  cache_sweeper :product_sweeper

  def index
    @products = Product.all
  end

end
</pre></div>

<h4 id="sql-caching">5. Кэширование <span class="caps">SQL</span></h4>
<p>Кэширование запроса это особенность Rails, кэширующая результат выборки по каждому запросу. Если Rails встретит тот же запрос (query) на протяжения текущего запроса (request), он использует кэшированный результат, вместо того, чтобы снова сделать запрос к базе данных.</p>
<p>Например:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ProductsController < ActionController

  def index
    # Запускаем поисковый запрос
    @products = Product.all

    ...

    # Снова запускаем тот же запрос
    @products = Product.all
  end

end
</pre></div>
<p>Второй раз к базе данных обращен тот же запрос, но он фактически не затронет базу данных. Результат, возвращенный первый раз от запроса, сохранится в кэше запроса (в памяти), и во второй раз будет получен из памяти.</p>
<p>Однако, важно отметить, что кэши запросов создаются в начале экшна и уничтожаются в конце этого экшна, и поэтому сохраняются только на протяжении экшна. Если хотите хранить результаты запроса более долгий период, воспользуйтесь низкоуровневым кэшированием Rails.</p>
