h1. Фильтры

<p>Фильтры это методы, которые запускаются до, после или &#8220;вокруг&#8221; экшна контроллера.</p>
<p>Фильтры наследуются, поэтому, если вы установите фильтр в <tt>ApplicationController</tt>, он будет запущен в каждом контроллере вашего приложения.</p>
<p>Предварительные фильтры могут прерывать цикл запроса. Обычный предварительный фильтр это, например, тот, который требует, чтобы пользователь был авторизован для запуска экшна. Метод фильтра можно определить следующим образом:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ApplicationController &lt; ActionController::Base
  before_filter :require_login

private
  def require_login
    unless logged_in?
      flash[:error] = &quot;You must be logged in to access this section&quot;
      redirect_to new_login_url # прерывает цикл запроса
    end
  end

  # метод logged_in? просто возвращает true если пользователь авторизован
  # и false в противном случае Это так называемый &quot;booleanizing&quot;
  # к методу current_user, который мы создали ранее, применили двойной ! 
  # оператор. Отметьте, что это не обычно для Ruby и не рекомендуется, если
  # Вы действительно не хотите конвертировать что-либо в true или false.
  def logged_in?
    !!current_user
  end
end
</pre></div>
<p>Метод просто записывает сообщение об ошибке во flash и перенаправляет на форму авторизации, если пользователь не авторизовался. Если предварительный фильтр рендерится или перенаправляет, экшн не запустится. Если есть дополнительные фильтры в очереди, они также прекращаются.</p>
<p>В этом примере фильтр добавлен в <tt>ApplicationController</tt>, и поэтому все контроллеры в приложении наследуют его. Это приводит к тому, что всё в приложении требует, чтобы пользователь был авторизован, чтобы быть использовано. По понятным причинам (пользователь не сможет зарегистрироваться в первую очередь!), не все контроллеры или экшны должны требовать его. Вы можете не допустить запуск этого фильтра перед определенными экшнами с помощью <tt>skip_before_filter</tt>:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class LoginsController &lt; Application
  skip_before_filter :require_login, :only =&gt; [:new, :create]
end
</pre></div>
<p>Теперь, экшны <tt>LoginsController</tt> <tt>new</tt> и <tt>create</tt> будут работать как раньше, без требования к пользователю быть зарегистрированным. Опция <tt>:only</tt> используется для пропуска фильтра только для этих экшнов, также есть опция <tt>:except</tt>, которая работает наоборот. Эти опции могут также использоваться при добавлении фильтров, поэтому можете добавить фильтр, который запускается только для выбранных экшнов в первую очередь.</p>

<h4 id="after-filters-and-around-filters">1. Последующие фильтры и охватывающие фильтры</h4>
<p>В дополнение к предварительным фильтрам, можете запустить фильтры после того, как экшн был запущен, или и до, и после. Последующий фильтр подобен предварительному, но, поскольку экшн уже был запущен, у него есть доступ к данным отклика, которые будут отосланы клиенту. Очевидно, последующий фильтр не сможет остановить экшн от запуска.</p>
<p>Охватывающие фильтры ответственны за запуск экшна с помощью yield, подобно тому, как работает Rack middlewares.</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ChangesController < ActionController::Base
  around_filter :wrap_in_transaction, :only => :show

  private

  def wrap_in_transaction
    ActiveRecord::Base.transaction do
      begin
        yield
      ensure
        raise ActiveRecord::Rollback
      end
    end
  end
end
</pre></div>
<p>Отметьте, что обертка охватывающих фильтров также рендерится. В частности, если в вышеуказанном примере вьюха сама начнет считывать из базы данных через скоуп или что-то еще, он это осуществит внутри транзакции, предоставив, таким образом, данные для предварительного просмотра..</p>
<p>В них можно не вызывать yield и создать отклик самостоятельно, в этом случае экшн не будет запущен.</p>

<h4 id="other-ways-to-use-filters">2. Другие способы использования фильтров</h4>
<p>Хотя наиболее распространенный способ использование фильтров это создание private методов и использование *_filter для их добавления, есть два других способа делать то же самое.</p>
<p>Первый это использовать блок прямо в методах *_filter. Блок получает контроллер как аргумент, и вышеупомянутый фильтр <tt>require_login</tt> может быть переписан с использованием блока:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ApplicationController &lt; ActionController::Base
  before_filter do |controller|
    redirect_to new_login_url unless controller.send(:logged_in?)
  end
end
</pre></div>
<p>Отметьте, что фильтр в этом случае использует метод <tt>send</tt>, так как <tt>logged_in?</tt> является private, и фильтр не запустится в области видимости контроллера. Это не рекомендуемый способ применения такого особого фильтра, но в простых задачах он может быть полезен.</p>
<p>Второй способ это использовать класс (фактически, подойдет любой объект, реагирующий правильными методами) для управления фильтрацией. Это полезно для более сложных задач, которые не могут быть осуществлены предыдущими двумя способами по причине трудности читаемости и повторного использования. Как пример, можете переписать фильтр авторизации снова, использовав класс:</p>
<div class="code_container"><pre class="brush: ruby; gutter: false; toolbar: false">
class ApplicationController &lt; ActionController::Base
  before_filter LoginFilter
end

class LoginFilter
  def self.filter(controller)
    unless controller.send(:logged_in?)
      controller.flash[:error] = &quot;You must be logged in&quot;
      controller.redirect_to controller.new_login_url
    end
  end
end
</pre></div>
<p>Опять же, это не идеальный пример для этого фильтра, поскольку он не запускается в области видимости контроллера, а получает контроллер как аргумент. Класс фильтра имеет метод класса <tt>filter</tt>, который запускается до или после эшна, в зависимости от того, определен ли он предварительным или последующим фильтром. Классы, используемые как охватывающие фильтры, могут также использовать тот же метод <tt>filter</tt>, и они будет запущены тем же образом. Метод должен иметь <tt>yield</tt> для исполнения экшна. Альтернативно, он может иметь методы и <tt>before</tt>, и <tt>after</tt>, которые запускаются до и после экшна.</p>
